# StrataWP Quality & Accessibility System — Design Spec (Phase 3)

- **Date:** 2026-06-24
- **Branch:** `feat/quality-a11y` (off `main` @ `117b49d`)
- **Status:** Draft — awaiting review (revised after adversarial review)
- **Sub-project 3 of the framework-hardening program.** Comparison target is codenamed **Triple XXX**.

## 1. Goal

Make StrataWP's quality and accessibility posture **enforced across the whole codebase, not just the published core** — and prove it with two new CI gates that Triple XXX defines bars for but never runs:

1. **WordPress Coding Standards (PHPCS/WPCS) on _every_ PHP surface** — the three example themes _and_ the three CLI scaffold templates users actually generate from — not only `packages/core/src`.
2. **Automated front-end accessibility testing** — Playwright + axe-core against the rendered `examples/basic-theme` on a real WordPress (wp-env), gating on zero WCAG 2.1 A/AA violations.

North star (scorecard rematch): on the **Quality & a11y** dimension StrataWP moves from "WPCS on `core/src` only, zero front-end a11y testing" to "WPCS green across **all** PHP + zero serious/critical axe violations, **both required CI checks**." The thesis is consistent with Phases 1–2: Triple XXX _declares_ standards and a Lighthouse a11y bar but does not _enforce_ them in CI; StrataWP wins by enforcing, with evidence.

## 2. Current reality (verified by recon + spot-checked against the repo)

### 2a. PHP / WPCS

`packages/core/src` is linted (its own ratcheted ruleset, **0 err / 0 warn**). **Nothing else is.** The CI `php` job hard-codes `working-directory: packages/core`, so ~460 PHP files across the example themes and CLI templates get **zero** lint signal.

The base ruleset for themes will be **`WordPress-Extra`** (see §5 decision 3): it includes `WordPress-Core`'s formatting **plus the security/best-practice sniffs** (`WordPress.Security.*`, `WordPress.WP.*`) but **not** `WordPress-Docs` and **not** a configured `text_domain`. Measured debt under `WordPress-Extra` (vendor/node_modules/`*-generated.php` ignored), with `phpcbf` auto-fix capacity:

| Location | Errors | Warnings | Files | `phpcbf` auto-fixes | Manual residue |
| --- | --- | --- | --- | --- | --- |
| `examples/basic-theme` | 419 | 6 | 10 | 417 | ~2 (PSR-4 `FileName`) |
| `examples/store-theme` | 774 | 12 | 17 | 766 | `EscapeOutput` ×6, `GlobalVariablesOverride` ×4 |
| `examples/advanced-theme` | 1,339 | 138 | 15 | 1,457 | `EscapeOutput` ×2, `GlobalVariablesOverride` ×4 |

Key facts (verified):

- **~99% is mechanical.** The dominant sniff is `Generic.WhiteSpace.DisallowSpaceIndent` (~380 in basic alone; pattern/PHP authored with spaces, WPCS wants tabs). `phpcbf` clears it and the other formatting sniffs (short arrays, function-call signatures, statement alignment).
- **Residue after `phpcbf` is tiny and falls into two classes:**
  - **PSR-4 file-name** (`WordPress.Files.FileName.NotHyphenatedLowercase` + `InvalidClassFileName`, ×2 in basic) — the **same class core already ratchets** for PascalCase autoloaded files → suppress identically.
  - **Genuine security/correctness** — NOT `phpcbf`-fixable: `WordPress.Security.EscapeOutput.OutputNotEscaped` (store ×6, advanced ×2) and `WordPress.WP.GlobalVariablesOverride` (×4 each). These must be **fixed** (escaping) or **justified with a scoped `phpcs:ignore` + comment** (e.g. the legitimate `$content_width` theme idiom, if that's what the override is) — **never baselined away**. The build verifies which each one is.
- **`store` and `advanced` examples are byte-identical PHP to their `cli/templates` twins** (apart from machine-generated `inc/*-generated.php`), so 4 of 6 locations collapse to **2 canonical fixes**; `basic` has diverged (the example wires more components) and needs **2 passes**. The 6 CLI-template/example locations total **~5,000 violations, ~99% `phpcbf`-fixable**.
- **Core's full ruleset is NOT theme-reusable.** Pointed at `advanced` with `WordPress-Docs` + a configured `text_domain` it explodes (doc-comment noise on every file; **439 `TextDomainMismatch`** per theme from the legacy `frost` text domain — see §2b). Themes get a **`WordPress-Extra`, no-`text_domain`** variant; i18n enforcement is deferred (decision 4).
- **PHP QA deps live only in `packages/core`** (`squizlabs/php_codesniffer` 3.13.5, `wp-coding-standards/wpcs` ^3.1, `phpcompatibility/phpcompatibility-wp`). The installed `packages/core/vendor/bin/phpcs` already has `WordPress`, `WordPress-Core`, `WordPress-Extra`, `WordPress-Docs`, `PHPCompatibilityWP` — sufficient to lint every location without a second install.
- Minor correctness: `examples/basic-theme/composer.json` has a **duplicate `require` key** (second wins → silently drops the `php >= 8.1` constraint). Fix while in there.

### 2b. Accessibility / front-end testing

- 🚨 **Render blocker — legacy `frost` naming.** Content references the pattern namespace `frost/*` (6 files under `templates/` + 2 pattern files with nested refs, e.g. `patterns/page-pricing.php` → `frost/pricing-three-columns-dark`), but patterns register as `strata-basic/*`. `CLAUDE.md` documents yet a third spelling (`forge-basic/*`, ~line 422), and every theme also carries **439 `'frost'` text-domain tokens** in gettext calls (style.css declares `strata-basic`/`strata-store`/`strata-advanced`). Result: `<main>` renders **empty** on home/blog/single/search/archive/404 (header + footer use the correct slug). **Any axe run before this is fixed is meaningless.**
- Once content renders, the known axe failures on `examples/basic-theme` are: **no skip-link** in the FSE render (`bypass`); **no `<h1>`** on home, hero is `<h2>` (`page-has-heading-one` + `heading-order`); **duplicate `id="text-on-left-image-on-right"`** in `page-home` (`duplicate-id`); **"Get Started" `<a>` with no `href`** (`link-name` / not focusable); **`outline:none`** on form focus with no `:focus-visible` (`_forms.scss`); **links non-underlined and same color as body text** (`theme.json` `textDecoration:none`, `link-in-text-block`); **`aria-current`** wired only for the classic `wp_nav_menu`, not `core/navigation` block nav.
- **`packages/testing`** is already the Playwright home (`@stratawp/testing`, exports a `playwright` subpath, 25+ WP helpers, package-level `test:e2e`/`test:coverage` scripts) but has **no axe**; a **stale** `@playwright/test ^1.40.1` (lock resolves 1.57.0); `testDir: './e2e'` pointing at a directory that doesn't exist; **5 browser projects** (chromium/firefox/webkit/Mobile Chrome/Mobile Safari); `baseURL` already defaults to `:8888`; and a local-only `webServer` (CI-gated to `undefined`) wrongly aimed at `:3000` (Vite HMR). A `block-editor.spec.ts` exists under `examples/basic-theme/e2e/` and the `cli/templates/basic-theme/e2e/` twin — it tests the block _editor_ via `wpLogin`, **not** front-end a11y, and is collected by **no** config today. Playwright runs in **no** CI job.
- **The wp-env / Lighthouse boot from Phase 2 is the model to follow** (not "reuse verbatim" — see §4 WS-B). `perf.yml` boots its own wp-env on the **dev site `:8888`** via the `cli` service: build `@stratawp/vite-plugin` → (in the theme, with `COMPOSER_MIRROR_PATH_REPOS=1`) `composer install` + `pnpm build` → `wp-env start` → `wp-env run cli wp theme activate basic-theme`. `.wp-env.json` maps only `./examples/basic-theme` with `WP_DEBUG:false` (no custom `tests` config; there is no `:8889`/`tests-cli` wiring to rely on). `scripts/extract-critical.mjs` does a **single one-shot `fetch`** — it is **not** a readiness poller.
- **Doc drift:** root `package.json` defines **neither** `test:e2e` nor `test:coverage`, yet `CLAUDE.md` advertises both; `turbo.json` (Turbo **1.x**, `pipeline` key) has no `e2e` task. `CLAUDE.md` (~line 85) and `packages/core/src/Components/Accessibility.php:5` each contain a credit line **naming the competitor** — codename leaks in tracked files.

### 2c. What Triple XXX does here (the bar)

Triple XXX ships a WordPress-Coding-Standards `phpcs` config over its theme and an `a11y ≥ 0.9` Lighthouse posture, but **does not enforce either as a required CI gate**, and ships **no automated axe/Playwright front-end a11y testing**. The win is the same shape as Phase 2: **enforce what they only declare**, and add the automated a11y testing they lack.

## 3. Scope

**Two workstreams, one spec, one plan each, one PR each** (mirrors Phase 2). They are independent (PHP linting vs browser a11y), touch disjoint files, and have different gating maturity.

- **WS-A — WPCS across ALL PHP** ships **first** (largest, most mechanical, no Docker, fast feedback; closes the biggest CI blind spot and the real security gaps; a clean PHP tree de-risks WS-B).
- **WS-B — Playwright + axe-core front-end a11y** ships **second** (net-new harness; depends on Docker/wp-env and on the `frost` naming being fixed first).

**Out of scope:** `@stratawp/sync` (frozen); `explorer`/`headless`; Phases 4–5. **i18n enforcement** (configuring `WordPress.WP.I18n` + the cross-theme `frost` text-domain sweep) is deferred — WS-B fixes basic-theme's text domain as part of its `frost` reconciliation; store/advanced/template text-domain + I18n enforcement is a tracked follow-up. **PHPStan on themes** is excluded (Phase 3 = PHPCS/WPCS + axe). The **broad codename-leak sweep** (vite-plugin header, README credits, scss/readme headers across all themes) belongs to **Phase 5 (Doc truth-up)** — Phase 3 fixes only the leaks _in its own path_ and introduces no new ones. We will **not** flip `perf.yml`'s `continue-on-error` (Jon's manual TODO, gated on calibration).

## 4. Workstreams

### WS-A — WPCS across ALL PHP (PR #1)

1. **Per-theme ruleset.** Add a `phpcs.xml.dist` per theme location (six total — three examples + three CLI templates), each based on **`WordPress-Extra`** (formatting + security/best-practice; **no `WordPress-Docs`, no configured `text_domain`**). Carry over core's documented `<severity>0</severity>` ratchet for the PSR-4 file-name class only (`WordPress.Files.FileName.NotHyphenatedLowercase` + `InvalidClassFileName`); add others **only** if a measured, justified residue appears after `phpcbf`. `<config name="testVersion" value="8.1-"/>`, `minimum_wp_version` to match core. Scope `<file>` to the theme root; **exclude** committed `vendor/`, `node_modules/`, `dist/`, and `inc/*-generated.php` (Vite-plugin build artifacts — decision 5).
2. **`phpcbf` first, ratchet second.** Run `phpcbf` per location to clear the ~99% mechanical violations (incl. space→tab indent — **fix it, do not suppress it**); commit as one mechanical "chore" per location, **separate** from hand edits. Then the ratchet only covers real residue. _Verify `phpcbf` does not rewrite literal inline HTML (`T_INLINE_HTML`) or `<!-- wp:* -->` block-delimiter comments inside pattern files (spot-diff a pattern before/after; pattern files contain no heredocs — the risk is the inline-HTML regions)._
3. **Fix the security/correctness residue by hand** — `EscapeOutput` (store ×6, advanced ×2 — add proper `esc_*`); assess `GlobalVariablesOverride` (×4 each — if it's the standard `$content_width` theme idiom, document with a scoped `phpcs:ignore` + comment; else fix); re-check for any `MissingUnslash`/`SlowDBQuery` the full ruleset would catch and treat the same. **Never baseline these.** Also fix the `examples/basic-theme/composer.json` duplicate `require` key.
4. **Lint mechanism — light, root-script-driven.** Keep QA deps only in `packages/core`; lint each location via `packages/core/vendor/bin/phpcs --standard=<location>/phpcs.xml.dist`. Add a **root `package.json` `lint:php` script** that iterates all six locations (the CLI templates are **not** pnpm workspaces, so a per-workspace turbo task can't reach them — the root script is the source of truth). Optionally add a turbo `phpcs` task (under `turbo.json`'s **`pipeline` key — Turbo 1.x**) for the example-theme workspaces; do **not** name it `php` (collides with the CI job). The DoD-proving command is `pnpm lint:php` → exit 0.
5. **CI gate — add a job, don't rewrite the core one.** Leave the existing `php` job intact (phpcs + phpstan + phpunit on `packages/core`, matrix 8.1–8.4). Add a **new `php-themes` job** (single PHP version, e.g. 8.3 — phpcs is runtime-agnostic, `PHPCompatibility` covers 8.1–): `composer install` in `packages/core` (to provide the phpcs binary + WPCS), then `pnpm lint:php`. Theme locations get **phpcs only** (no phpstan/phpunit). **Turn the gate on per-location, in the same PR, as each location is cleaned** so a green gate locks in each gain.
6. **Keep the byte-identical pairs in sync.** Apply each canonical fix to both the `cli/templates/*` source **and** its `examples/*` twin (`store`, `advanced`). Fix `cli/templates/*` deliberately — it's what every new theme inherits, so cleaning it stops new debt at origin. `basic` diverges → two passes.

**Rollout order (cheapest gate first, never regress):** `cli/templates/basic` → `examples/basic` → `store` (one fix → both) → `advanced` (one fix → both; highest debt + the security items).

### WS-B — Playwright + axe-core front-end a11y (PR #2)

1. **Fix the legacy `frost` naming first (basic-theme).** Reconcile **all** `frost` references so pages render: the `frost/*` pattern namespace in the 6 templates + 2 nested pattern refs → `strata-basic/*`; the unregistered `frost-page` pattern category; and the 439 `'frost'` text-domain tokens → `strata-basic`. Reconcile the `forge-basic` spelling in `CLAUDE.md`. **Verify a repo-wide `grep "frost"` over `examples/basic-theme` returns zero**, then confirm `<main>` renders real markup on home/blog/single/search/archive/404 before writing any assertion.
2. **Harness in `packages/testing`.** Create `packages/testing/e2e/` (the `testDir` already points there) and put the front-end a11y specs there. Add `@axe-core/playwright`; centralize the scan in **one `AxeBuilder` fixture** (`test.extend`) so the tag set + any allowlist live in one audited place. Update `playwright.config.ts`: keep `baseURL` as the wp-env URL (`:8888`, already the default), **prune the project list to a single `chromium` project** for the a11y run (the 4 extra browser/mobile projects would fail under a single-engine browser install), and re-point or remove the local-only `:3000` `webServer` (it's already CI-gated off). The existing `block-editor.spec.ts` (editor, needs `wpLogin`) is **left in place and excluded from the a11y gate** — Phase 3's a11y job runs the front-end a11y specs only; wiring the editor spec into CI is out of scope.
3. **Boot wp-env on the dev site, modeled on `perf.yml` (not "verbatim").** The a11y job boots its **own** ephemeral wp-env, so the dev site `:8888` is fully isolated per-run — no separate test site needed. Steps: build `@stratawp/vite-plugin` → (in the theme, **`COMPOSER_MIRROR_PATH_REPOS=1`** — mandatory, or `vendor/stratawp/core` symlinks unresolvable in Docker → fatal → empty scan) `composer install` + `pnpm build` → `wp-env start` → `wp-env run cli wp theme activate basic-theme` → `wp-env run cli wp rewrite flush --hard` → **seed a published page (front page) + a post + assign a nav menu** via `wp-env run cli wp ...` → **poll for HTTP 200 readiness** with a **new** small retry/backoff helper (the existing `extract-critical.mjs` is a one-shot fetch, not a poller) → run the a11y project. Scan **unauthenticated** (no `wpLogin`); `.exclude('#wpadminbar')` defensively; scan after `document.fonts.ready` + a web-first `expect(locator).toBeVisible()` (not `networkidle`).
4. **Fix the a11y defects** in `examples/basic-theme` so the gate goes green: skip-link in the FSE header part/pattern (`bypass`); one `<h1>` + correct heading order on home; de-duplicate the `id`; real `href`s for the "Get Started" controls (or make them `<button>`); `:focus-visible` styling in `_forms.scss`; link affordance (underline or sufficient contrast) in `theme.json`; `aria-current` for `core/navigation`.
5. **axe gate.** `expect(results.violations).toEqual([])` (never forget the assert — `analyze()` never throws) for tags `['wcag2a','wcag2aa','wcag21a','wcag21aa']`. `best-practice` runs **advisory only** (separate, non-blocking). Every disabled rule carries a code comment + issue link; prefer `.exclude()`-ing a specific node over global `disableRules`. For color-contrast specifically, fix the `theme.json` palette rather than disabling the rule.
6. **CI — new workflow `a11y.yml`, hard-blocking from day one** (don't overload `perf.yml`, which is `continue-on-error`). Model on `perf.yml`'s wp-env flow; pin Node 20; `playwright install --with-deps chromium` (one engine — axe is browser-agnostic); cache only the pnpm store (Playwright advises against caching browser binaries); set a generous `wp-env start` timeout (first run pulls WordPress/MySQL images). Reporters `[['github'],['html'],['blob']]`; upload report `if: !cancelled()`, traces `if: failure()`. axe violations are deterministic, so unlike Lighthouse budgets this gate is **required immediately**.
7. **Reconcile docs & wiring.** Add a real root `test:e2e` script (turbo passthrough; package-level `test:e2e` already exists) and a turbo `e2e` task; **remove** the unimplemented `pnpm test:coverage` advertisement from `CLAUDE.md` (or add a real root script — recommend remove); append `playwright-report/` and `test-results/` to `.gitignore` (and confirm `.prettierignore` covers them). Update the `CLAUDE.md` Accessibility/Testing sections to the enforced reality. **Neutralize the codename leaks in this path** (`Accessibility.php:5`, `CLAUDE.md`) to a neutral "prior-art WordPress starter theme (GPL)" credit — decision 7. (Run `pnpm format` after `CLAUDE.md` edits — `**/*.md` is prettier-checked; `docs/superpowers/*` is in `.prettierignore` so this spec/plan won't trip `format:check`.)

**Pinned versions** (mutually compatible, Node 18–24): `@playwright/test ^1.58.2`, `@axe-core/playwright ^4.11.3` (bundles axe-core 4.11.x). Install via `pnpm --filter @stratawp/testing add -D`.

## 5. Key decisions (my call unless you redline)

1. **Split into two sequenced PRs** — WS-A (WPCS-all) then WS-B (axe) — under one spec, one plan each. Disjoint files; WS-A de-risks WS-B.
2. **Fix-then-ratchet, per location, gated incrementally** — run `phpcbf`, fix real residue, ratchet only the PSR-4 `FileName` class (as core does), switch CI on per location.
3. **`WordPress-Extra` base ruleset** (not `WordPress-Core`, not full `WordPress`) — catches the security sniffs that matter while excluding `WordPress-Docs` doc-noise; **no `text_domain` configured** so the legacy `frost` text domain doesn't self-block the gate with 439 `TextDomainMismatch` errors.
4. **i18n enforcement deferred** — WS-A does not configure `WordPress.WP.I18n`. WS-B fixes basic-theme's `frost` text domain as part of its naming reconciliation; cross-theme i18n enforcement is a tracked follow-up (re-enable `WordPress.WP.I18n` once text domains are clean).
5. **Path-exclude `inc/*-generated.php`** from theme rulesets (Vite-plugin output; hand-fixes get overwritten); fix the generator template in `packages/vite-plugin` only if trivial, else defer.
6. **Security sniffs are fixed (or scoped-`phpcs:ignore` with justification), never baselined.**
7. **Attribution = neutral credit** — replace the credit lines naming the competitor (in `Accessibility.php:5`, `CLAUDE.md`) with a neutral "prior-art WordPress starter theme (GPL)"; flag the wider sweep for Phase 5. _(Your decision.)_
8. **a11y gate = `examples/basic-theme` only, full WCAG 2.1 A/AA zero-violations, hard-blocking from day one**; store/advanced a11y deferred; `best-practice` advisory-only. _(Your decision.)_
9. **Light lint mechanism** — shared `packages/core/vendor/bin/phpcs` + per-theme ruleset driven by a root `lint:php` script (covers the non-workspace CLI templates); a new `php-themes` CI job runs phpcs-only over all six locations, leaving the core `php` job (phpcs+phpstan+phpunit) untouched.

## 7. Risks & mitigations

- **`COMPOSER_MIRROR_PATH_REPOS=1` is mandatory** on the composer-install step for the a11y job (as for perf). Without it `vendor/stratawp/core` symlinks outside the mounted theme dir → unresolvable in Docker → `autoload.php` fatal → white screen → empty scan.
- **Legacy `frost` naming** must be fully reconciled (namespace + category + text domain) before any axe assertion, or every number is meaningless. First task of WS-B; verified by `grep "frost"` → 0.
- **`phpcbf` diff size** — space→tab across ~50 pattern files × multiple themes is a huge, mechanical diff. Commit `phpcbf` output separately from hand edits; **verify `phpcbf` does not rewrite literal inline HTML / `wp:*` block-delimiter comments** in pattern files (`T_INLINE_HTML` is left untouched by the indent sniff — confirm with a before/after diff).
- **Security residue may include legitimate idioms** (`$content_width` `GlobalVariablesOverride`) — assess each; scoped-`phpcs:ignore` with a comment where genuinely correct, fix otherwise. Don't blanket-suppress the sniff.
- **`WordPress-Extra` without `text_domain`** is the deliberate choice to keep the gate green; if it surfaces stray i18n sniffs (e.g. `MissingTranslatorsComment`), fix or ratchet per-case — do not re-introduce `TextDomainMismatch` by configuring `text_domain` before the `frost` sweep.
- **color-contrast false positives** are the #1 WP-theme axe pitfall (disabled controls, gradient/pseudo bgs, text-shadow, visually-hidden text). Fix the `theme.json` palette or `.exclude()` the specific node — never blanket-disable the rule.
- **Docker/wp-env in CI** — first run pulls images (set a generous `wp-env start` timeout); the a11y job is isolated in its own workflow on its own ephemeral wp-env. `a11y.yml` pins Node 20; local dev on Node 25 is outside the tested 18/20/22/24 matrix (not fatal, but note it).
- **Codename discipline** — neutral branch `feat/quality-a11y`; never quote the recon-clone path; the **untracked** stray `docs/superpowers/plans/2026-04-26-*.md` plan (its filename embeds the competitor name) must **never** be `git add`ed — stage files explicitly, never `git add -A`/`git add .`.
- **`CLAUDE.md` is format-gated** by prettier (`**/*.md`) — run `pnpm format` after editing it; `docs/superpowers/*` is in `.prettierignore` so the spec/plan won't trip `format:check`.
